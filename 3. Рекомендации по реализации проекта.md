# Veles — рекомендации по реализации проекта

> Документ описывает организационные условия, необходимые для успешной реализации проекта Veles, и роль руководителя проекта в соответствии с методологией PMBOK.

См. также: [PROJECT.md](1.%20Описание%20проекта.md) — описание продукта и технического плана; [INTEGRATION_DIADOC.md](5.%20Интеграция%20с%20Diadoc.md), [INTEGRATION_AVANKOR.md](6.%20Интеграция%20с%20Аванкор.md), [INTEGRATION_SPEC_DEP.md](8.%20Интеграция%20со%20Спецдепозитарием.md) и [INTEGRATION_BANK_CLIENT.md](7.%20Интеграция%20с%20Банк-клиентом.md) — интеграции.

## 1. Что нужно для реализации проекта

Проект Veles — это не только разработка программного обеспечения, но и изменение бизнес-процессов бухгалтерии и бэк-офиса УК ПИФ. Для успешной реализации необходимы:

| Роль / ресурс                    | Назначение                                                                                      |
| -------------------------------- | ----------------------------------------------------------------------------------------------- |
| **Руководитель проекта**         | Управление сроками, требованиями, рисками и коммуникацией между бизнесом и разработкой          |
| **Заказчик / владелец продукта** | Представитель бизнеса (бухгалтерия, сопровождение операций), принимающий решения по функционалу |
| **Команда разработки**           | Прототип: Streamlit · Prod: FastAPI + React; интеграции (Diadoc, Аванкор, Спецдеп, банк) |
| **Представители интеграций**     | Специалисты по Diadoc и 1С/Аванкор для настройки API и HTTP-сервисов                            |
| **Пользователи-пилоты**          | Помощник бухгалтера, согласующие — для приёмочного тестирования на реальных документах          |

Без выделенного **руководителя проекта** высок риск расхождения ожиданий бизнеса и того, что будет построено, а также потери контекста при смене участников или затягивании сроков.

## 2. Роль руководителя проекта

**Руководитель проекта (РП)** — отдельная роль, не совмещаемая с ведущим разработчиком или заказчиком. В проектах автоматизации документооборота, как Veles, РП нужен по следующим причинам:

### Согласование требований

- Собирает и фиксирует требования от бухгалтерии, бэк-офиса, согласующих и IT.
- Разрешает противоречия (например, разный набор полей для типов документов, маршруты согласования по фондам).
- Ведёт реестр открытых вопросов (см. раздел 8 в [PROJECT.md](1.%20Описание%20проекта.md)) и добивается решений в срок.
- Контролирует, чтобы изменения требований проходили через формальное согласование, а не «устные договорённости».

### Ведение документации по проекту

- Поддерживает актуальность проектных документов: устав, план, требования, статусы, риски.
- Обеспечивает единый источник правды для команды и заказчика.
- Фиксирует решения совещаний, протоколы приёмки, журнал изменений.
- Передаёт знания при смене участников — без РП контекст «живёт в переписке и головах».

### Дополнительные обязанности в контексте Veles

- Координация доступа к Diadoc API и HTTP-сервису Аванкор.
- Планирование пилотного запуска на ограниченном наборе фондов/документов.
- Управление рисками (задержки интеграций, соответствие учётной политике, безопасность ЭЦП и персональных данных).
- Отчётность перед руководством УК о прогрессе, блокерах и бюджете.

## 3. Методология PMBOK — краткий обзор

**PMBOK** (*Project Management Body of Knowledge*) — свод знаний и стандартов по управлению проектами, публикуемый **PMI** (Project Management Institute). Это не жёсткая инструкция, а набор принципов, областей знаний и практик, адаптируемых под конкретный проект.

### Ключевые идеи (актуальная редакция PMBOK 7)

| Принцип | Смысл для Veles |
|---------|-----------------|
| **Ориентация на ценность** | Каждая функция (Diadoc → согласование → Аванкор) должна измеримо сокращать ручной труд |
| **Системное мышление** | Учитывать связь процессов: Diadoc, email-согласования, Аванкор, ~20 фондов |
| **Лидерство** | РП мотивирует участников и снимает организационные блокеры |
| **Непрерывное улучшение** | Прототип → пилот → доработка (в т.ч. OCR/VLM на следующих этапах) |
| **Адаптивность** | Гибкое планирование итераций при неполной ясности по API Аванкор |

### Области производительности (Performance Domains)

PMBOK 7 выделяет восемь взаимосвязанных областей, в которых РП ведёт проект:

1. **Инициация** — обоснование проекта, заинтересованные стороны.
2. **Планирование** — объём, сроки, ресурсы, риски.
3. **Работа** — выполнение запланированных задач.
4. **Команда** — роли, коммуникация, вовлечённость.
5. **Заинтересованные стороны** — ожидания бухгалтерии, IT, руководства.
6. **Неопределённость** — риски интеграций и изменений требований.
7. **Поставка** — готовые инкременты (этапы из [PROJECT.md](1.%20Описание%20проекта.md)).
8. **Измерение** — прогресс, отклонения, качество.

### Группы процессов (классическая модель PMBOK 6)

```mermaid
flowchart LR
    I[Инициация] --> P[Планирование]
    P --> E[Исполнение]
    E --> M[Мониторинг и контроль]
    M --> E
    M --> C[Завершение]
```

Для Veles РП циклически проходит эти фазы: уточняет требования → планирует этап → команда реализует → приёмка и корректировка плана.

## 4. Документы руководителя проекта (по методологии PMBOK)

Ниже — основной набор документов, которые готовит и ведёт РП. Не все нужны в полном объёме на этапе прототипа; для Veles отмечены приоритеты.

### 4.1. Инициация

| Документ | Содержание | Приоритет для Veles |
|----------|------------|---------------------|
| **Устав проекта** (*Project Charter*) | Цели, границы, ключевые результаты, спонсор, полномочия РП, высокоуровневые риски и допущения | **Обязательно** |
| **Реестр заинтересованных сторон** (*Stakeholder Register*) | Список участников (бухгалтерия, согласующие, 1С, Diadoc), роли, интересы, способ коммуникации | **Обязательно** |

### 4.2. Планирование

| Документ | Содержание | Приоритет для Veles |
|----------|------------|---------------------|
| **План управления проектом** (*Project Management Plan*) | Сводный план: как проект будет выполняться, контролироваться и закрываться | **Обязательно** |
| **Описание содержания** (*Scope Statement*) | Что входит в прототип и что явно исключено (например, OCR — этап 6) | **Обязательно** |
| **Иерархическая структура работ** (*WBS*) | Декомпозиция на этапы 1–6 из [PROJECT.md](1.%20Описание%20проекта.md) | **Рекомендуется** |
| **Реестр требований** (*Requirements Documentation*) | Функциональные и нефункциональные требования, трассировка к этапам | **Обязательно** |
| **Календарный план / диаграмма Ганта** | Сроки этапов, зависимости (Diadoc до согласования, Аванкор после) | **Рекомендуется** |
| **План управления рисками** + **реестр рисков** | Риски, вероятность, влияние, меры реагирования | **Обязательно** |
| **План коммуникаций** | Совещания, статусы, каналы (чат, email, протоколы) | **Рекомендуется** |
| **План управления качеством** | Критерии приёмки этапов, сценарии тестирования на реальных PDF | **Рекомендуется** |
| **План управления изменениями** | Как фиксируются и утверждаются изменения требований | **Обязательно** |
| **Бюджет / оценка ресурсов** | Трудозатраты разработки, лицензии, инфраструктура | По необходимости |

*План управления проектом* обычно включает ссылки на дочерние планы: содержание, риски, коммуникации, качество, закупки и др.

### 4.3. Исполнение

| Документ | Содержание | Приоритет для Veles |
|----------|------------|---------------------|
| **Запросы на изменение** (*Change Requests*) | Формализованные предложения по изменению объёма или сроков | **Обязательно** |
| **Протоколы совещаний** | Решения по требованиям, интеграциям, приёмке | **Обязательно** |
| **Отчёты о выполнении работ** | Что сделано за итерацию, что в работе | **Рекомендуется** |

### 4.4. Мониторинг и контроль

| Документ | Содержание | Приоритет для Veles |
|----------|------------|---------------------|
| **Отчёты о статусе проекта** | Прогресс, риски, блокеры, прогноз сроков | **Обязательно** |
| **Журнал изменений** (*Change Log*) | История утверждённых и отклонённых изменений | **Обязательно** |
| **Реестр проблем** (*Issue Log*) | Текущие блокеры (доступ к API, маппинг полей Аванкор) | **Обязательно** |
| **Отчёты о качестве / приёмке** | Результаты тестирования этапа | **Рекомендуется** |

### 4.5. Завершение

| Документ | Содержание | Приоритет для Veles |
|----------|------------|---------------------|
| **Акт приёмки / отчёт о закрытии** | Подтверждение достижения целей прототипа или этапа | **Обязательно** |
| **Извлечённые уроки** (*Lessons Learned*) | Что сработало, что улучшить перед масштабированием на 20 фондов | **Рекомендуется** |
| **Архив проектной документации** | Финальные версии требований, планов, инструкций для пользователей | **Рекомендуется** |

## 5. Минимальный набор для старта Veles

Если ресурсы ограничены, РП должен подготовить как минимум:

1. **Устав проекта** — согласованный с руководством УК.
2. **Реестр заинтересованных сторон** — кто принимает решения по Diadoc, Аванкор, согласованиям.
3. **Реестр требований** — на базе раздела 5 [PROJECT.md](1.%20Описание%20проекта.md), с отметками «согласовано / уточнить».
4. **Реестр рисков и открытых вопросов** — объединить с чек-листом из раздела 8 PROJECT.md.
5. **План этапов с датами** — этапы 1–5 с ответственными.
6. **Еженедельный статус-отчёт** — прогресс, блокеры, решения.

Остальные документы наращиваются по мере перехода от прототипа к пилоту и промышленной эксплуатации.
